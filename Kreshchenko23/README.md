# Звіт з практичної роботи №23
## Індивідуальний проєкт. Проєктування даних: сутності, модель стану та збереження

### 1. Зв'язок із попередніми етапами (ПР21–ПР22)
* **Концепція (ПР21):** Застосунок для тренування зорової пам'яті «Memory Training» на основі пошуку парних карток із виведенням когнітивних довідок.
* **Інтерфейс (ПР22):** Реалізовано перемикання екранів через `QStackedWidget` (Головне меню -> GameWidget -> StatsWidget) та додано механіку відстеження кількості ігор.
* **Ціль ПР23:** Спроектувати модель даних, яка дозволить надійно зберігати результати ігор між запусками програми.

### 2. Основні сутності застосунку
Для забезпечення поточного MVP-функціоналу застосунок оперує такими об'єктами:
1. **GameSession (Ігрова сесія)** — основна сутність для довготривалого збереження. Описує кожну завершену гру.
2. **Card (Ігрова картка)** — допоміжна внутрішня сутність логіки для відстеження стану на полі.

### 3. Поля сутностей

#### Назва сутності: GameSession
* `id` (тип `int`, обов'язкове) — унікальний номер сесії в історії.
* `pairsCount` (тип `int`, обов'язкове) — кількість знайдених пар (в MVP строго 8).
* `gamesInSession` (тип `int`, обов'язкове) — кількість ігор, проведених безперервно під час цієї сесії.
* `date` (тип `QString`, обов'язкове) — дата завершення гри у форматі ISO 8601.

#### Назва сутності: Card (Тимчасова, існує в межах об'єкта GameLogic)
* `id` (тип `int`, обов'язкове) — індекс картки на полі (від 0 до 15).
* `icon` (тип `QString`, обов'язкове) — текстовий символ картки ("A", "B", "C" тощо).
* `isMatched` (тип `bool`, обов'язкове) — чи знайдено для картки пару.

### 4. Идентифікатори та зв'язки
* **Тип ID:** Для `GameSession` використовується числовий `id` з автоінкрементом. При завантаженні файлу програма визначає найбільший наявний ID і наступному запису присвоює `maxId + 1`.
* **Зв'язки:** В межах MVP сутності мають плоску структуру. Зв'язок один до багатьох закладено архітектурно на майбутнє через JSON-масив, але зараз він спрощений до єдиного загального профілю користувача.

### 5. Модель стану застосунку (State Model)
* **Постійні дані (Основні):** Загальний лічильник `m_totalGamesPlayed`, який відображається у `StatsWidget`, та масив історій завершених ігор.
* **Стан сесії (Тимчасовий):** Поточний індекс `QStackedWidget` (`0` — меню, `1` — гра, `2` — статистика), прапорець `isProcessing` для блокування повторних кліків, індекси відкритих карток `firstClickedIdx` та `secondClickedIdx`.
* **Налаштування:** Поточний стан лічильника `gamesPlayedInSession` для контролю втоми користувача та виведення попереджень про ефект відстані.

### 6. Спосіб збереження даних
* **Формат:** **JSON** (файл `user_stats.json`).
* **Обґрунтування:** JSON забезпечує легке читання, гнучку структуру для додавання нових полів у майбутніх ПР та має вбудовану підтримку в Qt6 через модулі `QJsonDocument`, `QJsonObject` та `QJsonArray`.
